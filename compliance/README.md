# PhiSQL conformance suite

This directory is the PhiSQL conformance test suite: a body of test cases and a
runner that decide whether an implementation correctly compiles PhiSQL to the
Phileas redaction policy schema. It is the objective basis for the claim
"PhiSQL-conformant."

PhiSQL is an open specification (see [`../spec/`](../spec/)) under a permissive
license, so anyone may implement it. The **name** "PhiSQL" is a registered
trademark of Philterd, LLC, reserved for implementations that pass this suite.
The suite is therefore the load-bearing piece of the governance model: without a
shared pass/fail oracle, "conformant" would be a marketing word; with one, it is
a verifiable claim about a specific spec version.

> **Status.** The suite currently lives inside the `philterd/phisql` repository
> so it can evolve in lockstep with the spec and the reference implementation. It
> is designed to be lifted into a standalone `philterd/phisql-conformance`
> repository later without change: the cases are plain data and the runner has no
> dependency on this repository's layout beyond the default adapter path.

## What "conformant" means

An implementation is conformant with PhiSQL v1.0 if, for every case in this
suite, it:

- compiles each **accept** case to exactly the expected Phileas JSON policy;
- accepts each **parse-only** case without error;
- rejects each **reject** case, at the correct layer (a grammar error versus a
  semantic error).

Conformance is claimed against a specific spec version. The cases here target
[`spec/v1.0`](../spec/v1.0).

## The adapter contract

The runner is implementation-neutral. It never calls Java, or any particular
compiler, directly. Instead it drives an **adapter**: a command that wraps the
implementation under test. The adapter is invoked as

```
<adapter> <path-to-.phisql-file>
```

and must obey this contract:

| Outcome | stdout | exit code |
|---|---|---|
| Compiled successfully | the Phileas JSON policy | `0` |
| Parse / grammar error | (ignored) | `2` |
| Semantic / catalog error (parsed, but not a valid policy) | (ignored) | `3` |

The two distinct reject codes let the suite assert not merely that an invalid
input is rejected, but that it is rejected at the right layer: a malformed token
must fail to *parse*, while a well-formed policy that names an entity the catalog
does not define must fail to *compile*.

To certify your implementation, write a small adapter script that maps your
compiler's behavior onto this contract and point the runner at it:

```sh
python3 run.py --adapter /path/to/your-adapter.sh
```

The reference adapter, [`adapters/reference.sh`](adapters/reference.sh), is the
worked example.

## Case taxonomy

Each case is a `.phisql` file. Its expectations are determined by where it lives
under [`cases/`](cases/):

```
cases/
  accept/        compile must succeed (exit 0) and produce JSON equal to the
                 sibling <name>.json (compared as parsed JSON, so key order and
                 whitespace are irrelevant)
  parse-only/    must be accepted (exit 0); output is not compared
  reject/parse/      must fail to parse (exit 2)
  reject/semantic/   must parse but fail to compile (exit 3)
```

`parse-only` exists for the discovery verbs (`FIND PII`, `DISCOVER ENTITIES`,
`SCAN`, `SELECT ... FROM findings`). They are part of the v1.0 grammar but target
a separate discovery-query schema rather than a Phileas policy, so the suite
checks that they parse without pinning a compiled output.

A case may carry an optional sibling `<name>.reject` file (JSON,
`{"messageContains": "..."}`) to additionally assert that the adapter's stderr
contains a substring.

The expected `.json` for accept cases is **generated by the reference
implementation** (see "Regenerating expected output" below). The reference
implementation is, by construction, the conformance oracle for v1.0; the same
contract is already enforced by the reference test suite, which compiles every
[`spec/v1.0/examples`](../spec/v1.0/examples) file and compares it against a
committed JSON.

## Running the suite

Prerequisites: Python 3, plus whatever your implementation needs. The reference
adapter additionally needs a JDK (the reference targets Java 25) and Maven.

Build the reference CLI once, then run:

```sh
# from the repository root
(cd reference && mvn -Pcli -DskipTests package)   # builds reference/target/phisql-<version>-cli.jar
python3 compliance/run.py
```

Useful flags:

```sh
python3 compliance/run.py -v                 # print every case, not just failures
python3 compliance/run.py --filter strategies   # run a subset by path substring
python3 compliance/run.py --adapter ./my.sh  # test a different implementation
```

The runner exits non-zero if any case fails, so it drops straight into CI.

## Adding cases

1. Drop a `.phisql` file under the directory for its expected outcome
   (`accept/<group>/`, `parse-only/`, `reject/parse/`, or `reject/semantic/`).
2. For an accept case, generate its expected JSON from the reference oracle:

   ```sh
   python3 compliance/run.py --bless
   ```

   `--bless` (re)writes the `.json` for every accept case by compiling it with
   the configured adapter. Only ever bless from the reference implementation;
   blessing from an implementation under test would defeat the purpose.
3. Run the suite and confirm it is green.

Accept cases that include a `POLICY` declaration must use a filename that matches
the declared name (hyphens and underscores are equivalent, so
`hipaa-safe-harbor.phisql` matches `POLICY hipaa_safe_harbor`). Most cases omit
`POLICY` entirely, since the policy name never appears in the compiled JSON; the
naming rule itself is exercised by the cases under `accept/policy` and
`reject/semantic/name-mismatch`.

## Known underspecified areas

A conformance suite should only assert behavior the spec defines unambiguously.
One area is currently looser in the reference implementation than the catalog
text suggests, so the suite deliberately does **not** pin it:

- **`STATIC_REPLACE` without a `value`.** The catalog marks `value` required, but
  the reference compiler does not enforce it at compile time.

These are tracked as spec/implementation gaps. When the contract is tightened (an
RFC plus a reference change), matching cases should be added here.
